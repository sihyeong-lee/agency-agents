# Elvis 플랫폼 설계 문서

이 문서는 AI Canvas 애플리케이션 페이지를 `Elvis`라는 **플랫폼 사이트**로 배포할 때의 구조 설계 문서다.

중요한 전제:

- `노무사 챗봇`은 Elvis 전체 플랫폼의 **한 개 메뉴**다.
- 즉 화면 전체를 챗봇처럼 만들지 않고, **플랫폼 홈 -> 메뉴 진입 -> 상담/근거** 구조로 간다.
- 시각 스타일과 세부 미장센은 Claude에게 맡기고, 이 문서는 **구조 / 노드명 / 설정값 / 연결 기준**만 다룬다.
- 실제 AI Canvas 애플리케이션 페이지에서는 **페이지 노드가 메뉴와 연결**되고, 샌드박스/에이전트는 그 페이지 노드 안에 들어간다고 이해하면 가장 안전하다.
- 공개용 Elvis에서는 `에이전트 UI`를 직접 노출하지 않고, `샌드박스_질문입력_Elvis`를 질문 입력 UI로 쓰는 것을 권장한다.
- 중요한 구분: `샌드박스_질문입력_Elvis`는 **에이전트 자체가 아니라 입력 화면**이다. 실제 추론과 답변 생성은 뒤의 `3`, `19A`, `19B` 같은 `에이전트 프롬프트` 노드가 맡는다.

---

## 1. Elvis 이름과 브랜드 해석

- 이름: `Elvis`
- 의미:
  - `el` = 전체
  - `vis` = wise
- 해석:
  - 회사 안에 흩어져 있는 지식, 규정, 판례, 사례, 실무 문안을 한 곳에 모으고
  - 필요한 순간에 **현명하게 꺼내 쓰는 플랫폼**

즉 Elvis는 단순 챗봇이 아니라 아래를 모두 포괄하는 지식 허브다.

1. 질문에 답한다.
2. 근거를 보여준다.
3. 규정/사례/문안을 다시 찾아 쓸 수 있게 한다.
4. 앞으로 다른 업무 도구도 같은 플랫폼 안에 추가할 수 있다.

---

## 2. 플랫폼 기본 구조

Elvis는 아래 메뉴 구조를 기본으로 한다.

```text
Elvis
├─ 홈
├─ 노무상담
├─ 근거
├─ 지식문서
└─ 작성도구
```

초기 배포는 아래 3개만 먼저 활성화해도 된다.

1. `홈`
2. `노무상담`
3. `근거`

`지식문서`, `작성도구`는 틀만 먼저 만들어 두고 실제 기능은 나중에 붙여도 된다.

## 5. 페이지 노드와 메뉴 연결 구조

애플리케이션 페이지는 아래처럼 이해하면 된다.

```text
애플리케이션 페이지
-> 페이지_Elvis_홈
   -> 샌드박스_Elvis_홈

애플리케이션 페이지
-> 페이지_Elvis_노무상담
   -> 샌드박스_질문입력_Elvis
   -> 샌드박스_답변뷰어

애플리케이션 페이지
-> 페이지_Elvis_근거
   -> 샌드박스_근거뷰어

애플리케이션 페이지
-> 페이지_Elvis_지식문서
   -> 샌드박스_지식문서허브

애플리케이션 페이지
-> 페이지_Elvis_작성도구
   -> 샌드박스_작성도구허브
```

즉 `홈`, `노무상담`, `근거`, `지식문서`, `작성도구` 각각에 **페이지 노드 1개씩** 만든다고 생각하면 된다.

## 6. 페이지 노드 공통 설정값

페이지 노드 공통 권장값:

1. 노드 타입: `페이지`
2. 출력 포트: `페이지`
3. 크기 맞춤: `자동 맞춤`
4. 플로우 노드 설정: `사용하지 않음`

이유:

- `자동 맞춤`이 첫 배포에서 가장 안전하다.
- 콘텐츠가 잘리기보다 화면 안에 들어오게 맞추는 쪽이 안정적이다.
- 지금 구조는 버튼 노드/리스트 노드/아이콘 노드의 클릭 이벤트로 전체 플로우를 제어하지 않는다.
- 메뉴 이동은 홈 샌드박스 안의 JavaScript `changeApplicationMenu()`로 처리하고, 질문 실행은 공개 메뉴에서는 `샌드박스_질문입력_Elvis`, 내부 테스트에서는 `에이전트_법률상담입력_디버그`가 맡는다.
- 따라서 `플로우 노드 설정`은 처음에는 비워 두는 것이 맞다.

`플로우 노드 설정`을 나중에 쓰는 경우:

- 버튼 노드 클릭 시 특정 메뉴로 이동하고 싶을 때
- 리스트 항목 클릭 시 특정 노드를 실행하고 싶을 때
- 아이콘 노드 클릭 이벤트를 메뉴 이동이나 노드 실행에 연결하고 싶을 때

즉 지금 MVP에서는 **페이지 노드의 플로우 설정은 건드리지 않는다.**

## 8. 질문이 실제로 흘러가는 순서

사용자가 Elvis 사이트에서 `노무상담` 메뉴로 들어가 질문하면 아래 순서로 실행된다.

```text
샌드박스_질문입력_Elvis
-> 3. 에이전트 프롬프트_질의정규화
-> 4. 파이썬_정규화파싱
-> 5. 파이썬_URL생성
-> 공식 / 내부구조화 / 내부문서 / 웹 branch
-> 18M2
-> 19A
-> 19B
-> 20 -> 21 -> 22/23
```

즉 Elvis 플랫폼에서도 공개용 질문 경로는 `샌드박스_질문입력_Elvis -> 3`부터 시작한다고 보면 된다. 내부 테스트에서만 `에이전트_법률상담입력_디버그 -> 1` 경로를 유지해도 된다.

   - `페이지_Elvis_근거`
   - `페이지_Elvis_지식문서`
   - `페이지_Elvis_작성도구`
5. 각 페이지 노드의 크기 맞춤을 `자동 맞춤`으로 설정
6. 각 페이지 노드의 플로우 노드 설정은 비워 둠
7. 페이지 노드 내부에 샌드박스/에이전트 배치
   - 공개용: `샌드박스_질문입력_Elvis`
   - 내부 테스트용 선택: `에이전트_법률상담입력_디버그`
8. `샌드박스_Elvis_홈` 코드 붙여넣기
9. `샌드박스_질문입력_Elvis`, `샌드박스_답변뷰어`, `샌드박스_근거뷰어` 코드 붙여넣기
10. `샌드박스_질문입력_Elvis(output) -> 3(dataset)` 선 연결
11. `샌드박스_답변뷰어`, `샌드박스_근거뷰어` 입력 포트 생성 순서 확인
12. 선 연결 확인
13. 페이지 제목 / 설명 / 로고 / OG 이미지 설정
14. 내부 테스트 사용자 초대
15. 테스트 질문 실행
16. 공개 범위 확정 후 배포

## 12. 처음부터 하지 말아야 하는 것

처음부터 아래를 한 번에 하지 않는다.

- 홈 화면 샌드박스가 직접 검색 그래프 전체를 제어하는 구조
- 샌드박스에서 `D6R_AIHUB`, `D6R_RULES` 같은 대용량 데이터를 직접 읽는 구조
- `홈`, `지식문서`, `작성도구`까지 모두 완성하려는 시도
- 페이지 노드의 `플로우 노드 설정`으로 전체 챗봇 플로우를 억지로 제어하려는 시도

처음에는 아래 기준만 지키면 된다.

1. `홈`은 플랫폼 소개 화면
2. `노무상담`은 질문 입력 + 답변 보기
3. `근거`는 근거 보기
4. 페이지 노드는 메뉴 컨테이너로만 사용
5. 공개용 질문 입력은 `샌드박스_질문입력_Elvis`, 내부 디버그만 `에이전트_법률상담입력_디버그`

이렇게만 해도 Elvis는 이미 **플랫폼처럼 보이고, 노무사 챗봇은 그 안의 한 메뉴**로 자연스럽게 들어간다.
